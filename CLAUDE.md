# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What This Is

A BepInEx mod for the game **Void Crew** (Unity, .NET Framework 4.7.2). It uses [VoidManager](https://github.com/Nihility-Shift/VoidManager) as a middleware layer and [Harmony](https://github.com/pardeike/Harmony) for runtime patching of game assemblies. The mod targets Thunderstore for distribution.

## Build Commands

```sh
dotnet build                   # Debug build
dotnet build -c Release        # Release build (also outputs a .zip for Thunderstore)
```

Custom NuGet sources are configured in the `.csproj` (BepInEx feed, samboy.dev feed) — no extra `nuget.config` needed.

## Key Files

| File | Purpose |
|------|---------|
| `VoidCrewTerminus/ReleaseFiles/PluginInfo.config` | **Primary metadata file.** Controls GUID, version, authors, description, dependencies. Edit this first when setting up the mod. |
| `VoidCrewTerminus/MyPluginInfo.cs` | Auto-generated by `VCMT.Thunderstore.DocBuilder` during build from `PluginInfo.config`. Do not edit by hand. |
| `VoidCrewTerminus/Plugins.cs` | BepInEx plugin entry point (`BepinPlugin`) and VoidManager registration (`VoidManagerPlugin`). |
| `VoidCrewTerminus/Patch.cs` | Harmony patch classes live here (or in additional files following the same pattern). |

## Architecture

**Entry flow:** The game loads BepInEx → BepinEx calls `BepinPlugin.Awake()` → `Harmony.CreateAndPatchAll(Assembly.GetExecutingAssembly())` automatically applies all `[HarmonyPatch]` classes in the assembly.

**Adding patches:** Create a class annotated with `[HarmonyPatch(typeof(TargetClass), "MethodName")]` and add `static void Prefix/Postfix/Transpiler` methods. Use `BepinPlugin.Log` for logging.

**Multiplayer:** `VoidManagerPlugin.MPType` is set to `MultiplayerType.All`, meaning all players in a session must have the mod installed. Change this in `Plugins.cs` if the mod becomes host-only or optional.

**Never touch `PhotonNetwork` from plugin `Awake()`.** BepInEx's chainloader finishes long before the game runs its own Photon setup (`Chainloader startup complete` precedes `Starting photon connect` in the logs). Referencing `PhotonNetwork` at load time forces PUN's static initializer to construct the `LoadBalancingClient` *before* the game applies its `ServerSettings`, which silently breaks matchmaking — the server list renders as raw region codes and lobby creation hangs on "connecting". This cost a bisect to find (Phase 8-A, `bd4c6f2`). Defer any PUN work — `AddCallbackTarget` included — behind VoidManager's room events (`JoinedRoom` / `HostCreateRoom` / `LeftRoom`); `VoidManager.Events.Instance` *is* safe at `Awake`. See `Features/Net/ForgeNetSync.Init()` for the pattern.

**Asset bundles / Blender files:** `Assets/` and `Blend/` directories are present for future use but currently empty.

## Mod Setup Checklist

Before shipping, fill in `PluginInfo.config`:
- `GUID`, `PluginName`, `PluginVersion`, `PluginDescription`
- `PluginOriginalAuthor`, `PluginAuthors`, `ThunderstoreID`
- `DependencyStrings` (already seeded with BepInEx + VoidManager)

The prebuild step will propagate these values into `MyPluginInfo.cs`, the Thunderstore `manifest.json`, and the `README.md`.

## Game Assembly Reference (`.voidcrew/`)


If the `.voidcrew/` directory is present in the project root, it contains decompiled game assemblies for local reference — use these instead of manually decompiling DLLs when researching types or methods to patch.

| Path | Contents |
|------|---------|
| `.voidcrew/decompiled/<Assembly>/` | Full decompiled C# source, one file per type |
| `.voidcrew/types-index.json` | 4,215 entries — each with `type`, `kind`, `namespace`, `assembly`, `file` |
| `.voidcrew/namespace-index.json` | 273 namespaces mapped to their member types |
| `.voidcrew/manifest.json` | Per-assembly checksums and file counts (compare to detect stale data after a game update) |
| `.voidcrew/dlls/` | Raw game DLLs used as the decompilation source |

**Quick lookup examples:**
```sh
# Find a type by name across all assemblies
grep -r "ShipModule" .voidcrew/decompiled/

# Query the index with jq
jq '.[] | select(.type == "ShipModule")' .voidcrew/types-index.json

# List all types in a namespace
jq '."CG.Ship"' .voidcrew/namespace-index.json
```

The key assemblies are `Assembly-CSharp` (main game logic, 3 312 files), `VoidCrewCommon`, `Assembly-CSharp-firstpass`, `PhotonUnityNetworking`, and `Opsive.UltimateCharacterController`.
