Upgrade Forge — System Design

v1 design VoidCrewTerminus BepInEx + Harmony + VoidManager

A mod-side overlay that extends every ship module's effective level from 3 to 10, gated by a new in-world Forge module that consumes relics, scales with sector progression, and rolls perks at upgrade time. All progression is per-run.

Context

Void Crew currently caps ship module "upgrades" at what players perceive as level 3. The vanilla mechanic is actually a BuildBox prefab swap — a handheld InteractiveCarryableModuleUpgrader replaces a placed module with a different prefab. There is no numeric level field on CellModule or any subclass; the only piece of the game with a real numeric tier is the Fabricator (FabricatorModule.currentTier, capped by the FabricatorUpgradeData.FabricatorTiers count = 3 in the shipped ScriptableObject).

This mod extends module progression to levels 4–10 by adding a new in-world ship module — the Upgrade Forge — and tracking per-module level + perk state entirely in mod-side overlay state, leaving vanilla L1–3 BuildBox-swap untouched. The mod also introduces three connected roguelike mechanics: Tiered Relics, Cursed Relics with Maintenance Burden, and Sector Escalation. All progression is per-run only (resets on ship death), matching Void Crew's existing roguelike loop.

Why an overlay, not a data extension Investigation of the reference-only Assembly-CSharp.dll (NuGet cache, no live install) confirmed: weapons, shields, reactors, and other modules have no level field. The vanilla L1/L2/L3 weapon "progression" is three different prefabs that the BuildBox swap system transitions between. Extending that would require authoring 7 new prefabs per weapon. The overlay approach instead applies stat modifiers to existing prefabs via Harmony patches, with per-instance state in a ConditionalWeakTable<CellModule, ForgeModuleState>.

System Summary (one paragraph)

Every ship module's effective upgrade cap extends from L3 to L10 via a mod-side overlay. Levels 4–10 are performed at a new in-world ship module, the Upgrade Forge. To upgrade a module, the crew (a) ensures the Forge's capacity ≥ the upgrade's relic cost, (b) physically inserts the required number of relics into the Forge while it's targeted at the module, and (c) commits — the host rolls a perk if the inserted relic tier matches an empty perk slot on the target. Three slots per module, tier-gated (any / Rare+ / Legendary). Perks come from a sparse matrix: a category-wide default pool, plus signature perks tied to specific flagship relics. Some relics are Cursed and roll downside perks that manifest as Maintenance Burden. The Forge's own level is driven by a Forge Meter that fills from sector progression (passive) and alloy spending at a Fabricator-style terminal (active) — making the Forge a genuine investment commitment, not a guaranteed milestone. Sector Escalation scales enemy threat and loot quality together per jump. All progression resets on ship death. Host arbitrates all rolls; clients sync via VoidManager ModMessage.

Glossary

Upgrade Forge
A new installable ship module added by this mod. Holds relics in-world, performs upgrades on other modules, has its own level.
Forge Meter
A progression bar on the Forge. Fills from sector progression (passive) AND alloy spending at the Forge terminal (active). When full, the Forge gains a level. Each level requires more meter than the previous.
Forge Capacity
Number of relics the Forge can hold simultaneously, equal to its current level. Gates the max module-upgrade level (L10 requires capacity ≥ 4).
Module Level
A mod-side integer per CellModule instance, range L3 (vanilla baseline) → L10. Stored in ConditionalWeakTable<CellModule, ForgeModuleState>. Resets on run end.
Relic Tier
A new property added to existing game relics (~25 in vanilla, currently untiered). Tiers: Common / Rare / Legendary. Tier is a quality modifier, not a gate — any tier can upgrade any level.
Upgrade Quality
The per-level outcome modified by the relic tier used. Higher tier → bigger stat bump AND higher perk-roll chance.
Perk
A special modifier rolled at upgrade time (e.g. "Vampiric", "Overcharged"). Authored content, drawn from pools.
Perk Slot
A module has 3 perk slots, each tier-gated. Slot 1 = any tier, Slot 2 = Rare+, Slot 3 = Legendary only. Perks only roll when the upgrade's relic tier matches an empty slot.
Category Pool
Default perk list per module category (weapons / reactors / shields / engines / scanners). Any relic can roll perks from this pool.
Signature Perk
A perk tied to a specific relic identity. Only rolls when that exact relic is the one being inserted. ~5–10 flagship relics get signatures in v1.
Cursed Relic
A relic whose perk pool includes downside-bearing perks.
Maintenance Burden
The operational quirk attached to a module by a cursed perk — random shutoffs, increased maintenance interactions, etc. Not a stat penalty (clean modules at L10 remain clean).
Sector Escalation
Per-jump scaling. Each completed sector jump bumps enemy threat AND shifts relic drop tiers toward higher rarities.

Module Levels (L4–L10)

Levels 4–10 are mod-side state, applied on top of whatever vanilla state the module is in. The mod does not modify game data files. When a module is upgraded by the Forge, an entry is added to a static ConditionalWeakTable<CellModule, ForgeModuleState> keyed by the module instance. Stat modifications are applied via Harmony postfixes on the module's stat-calculation methods (specific methods are TBD pending live-game decompile — see Pre-Flight).

The state record:

class ForgeModuleState {
    int    Level;             // 3 (vanilla) → 10
    Perk[] Slots;             // up to 3, indexed [0]=any, [1]=Rare+, [2]=Legendary
    MaintenanceBurdenFlags Burden;
    StatModSet Accumulated;   // summed contribution of all upgrades + perks
}

Because state is per-run only, the table is cleared on VoidManager.Events.HostStartSession (host-side) and on ship-death events received from the game. No persistence — no save format.

Relics & Tiers

The game ships ~25 relics, currently untiered. The mod introduces a Tier property assigned in RelicTierData.cs:

Common Rare Legendary

Tier is a quality modifier, not a gate. Any tier can be inserted for any module level. Lower-tier insertions still produce the level-up, they just deliver smaller stat bumps and cannot fill higher-tier perk slots.

Drop-rate shaping

Drop rates are not fixed — they are biased per sector by Sector Escalation. Early sectors yield mostly Common; deep sectors yield a Rare/Legendary mix.

Perks & Slots

Each module has 3 perk slots, each tier-gated:

SlotRequired tier to fillNotes
Slot 1Any (Common+)First perk the module ever rolls.
Slot 2Rare or LegendaryA Common upgrade after slot 1 is full does not fill this slot — perk roll skips.
Slot 3Legendary onlyA fully-perked module requires at least one Legendary across its upgrade path.

The perk pool — sparse matrix

Perks are authored, drawn from a sparse two-axis matrix of (relic identity × module category):

  • Each module category (weapons / reactors / shields / engines / scanners) has a shared Category Pool of ~3 perks. Any relic can roll from it.
  • ~5–10 flagship relics additionally carry Signature Perks that only roll when that specific relic is the one inserted.

Gamble reveal

The roll is committed — there is no preview, no take-back. Player inserts the relic, the perk is rolled host-side, and the result is broadcast. This preserves classic roguelike tension and keeps the UI simple (no draft picker).

Roll math Roll-chance shape (v1 placeholders, all config-driven): Common → 15% chance to roll a slot-1-eligible perk · Rare → 40% chance to roll a slot-1 or slot-2-eligible perk · Legendary → 75% chance to roll any slot. Roll outcomes are computed only on the host; clients animate using a broadcast seed.

The Forge Itself

The Forge is a new installable ship module shipped as a custom prefab in Assets/voidcrewterminus.metem (loaded via the existing AssetLoader.cs pipeline). It has three interaction surfaces:

  1. Module socket — a single-payload socket that accepts the BuildBox of the module being upgraded, mirroring the Fabricator's RecycleSocket.Payload pattern. The target module is deconstructed first; the resulting BuildBox is dropped into the Forge socket for the duration of the upgrade. The Forge writes the new pending level via ForgeOverlayTable.SavePendingLevel(box.photonView.ViewID, level), and the existing ForgePersistPatch restores the level when the crew rebuilds the box elsewhere on the ship — so the Forge never touches the live CellModule's stats directly.
  2. Relic insertion — crew physically carries a relic and inserts it into the Forge. The Forge holds up to Forge Capacity relics simultaneously.
  3. Alloy terminal — crew spends alloys at a Fabricator-style terminal to push the Forge Meter (see below). Mirrors the existing Fabricator self-upgrade pattern.

Upgrade flow: deconstruct the target module → drop the resulting BuildBox into the Forge's module socket → insert the required number of relics (matching the cost curve for the desired level) → commit. The Forge consumes the spent relics, the host computes stat-mod + perk roll, the result broadcasts to all clients, and the BuildBox is released so the crew can rebuild it in-place at the new level.

Design clarification (2026-07-01) Original wording said "target a module → insert relics → commit" without specifying how the target is selected. The chosen model is the Fabricator-style single-payload socket: the module is physically deconstructed into a BuildBox first, and the BuildBox occupies the Forge's module socket during the upgrade. This lets the Forge reuse the existing deconstruct/reconstruct → ForgePersistPatch level-persistence bridge without any live-stat manipulation, and gives players a legible physical loop (carry the module in, carry it out).

Forge Meter — Investment Commitment

The Forge's own level is driven by a meter that fills from two parallel sources:

SourceContribution (v1 defaults)Type
Successful sector jump+20 per jumpPassive
Alloys at Forge terminal+1 per alloyActive

Each level requires more meter than the previous. Default curve: L1→L2 = 100, L2→L3 = 150, L3→L4 = 225, multiplying by 1.5× per level. This means a typical 5–6 sector run gets the Forge to L2–L3 passively; L10 module upgrades require alloy investment. That makes the Forge a real strategic decision: spend alloys on hull repairs / Fabricator upgrades / Forge level.

Sector Escalation

Each successful sector jump increments a DifficultyScalar on the ForgeMeterController:

  • Enemy stats scale (HP, damage, optionally density) via Harmony postfixes on enemy spawn/init paths.
  • Relic drop table shifts: at scalar 0, only Common; at scalar 3+, Common/Rare mix; at scalar 6+, Rare/Legendary mix.
  • Forge Meter gets its +20 contribution per jump (as above).

Effectively, pushing deeper makes the run harder and shifts loot toward higher tiers — tightening the risk-reward loop.

Cursed Relics & Maintenance Burden

~30% of vanilla relics are flagged IsCursed = true in RelicTierData.cs (config-driven default). When a cursed relic is used for an upgrade:

  • The stat bump still applies in full — there is no stat penalty.
  • The perk roll, if it occurs, draws from a cursed-augmented pool that pairs a beneficial perk with a Maintenance Burden flag.
Design intent Maintenance Burden is operational, not statistical. A burdened module still hits its damage targets, but it makes the crew work — random shutoffs, manual reset interactions, increased heat ticks, etc. The intent is for cursed perks to create emergent gameplay (crew scrambles to keep the Overcharged turret online during a fight), not flat power loss.

Example cursed perks (illustrative, v1 authoring lives in PerkPool.cs):

PerkBenefitBurden
Overcharged+30% damagePowers off for 2–4s every 30–90s
Vampiric5% damage healed to ship hullModule radiates heat; nearby modules tick warmer
Erratic+50% fire rate~5% chance per shot to require manual reset

Native-system Refinements

Three pieces of the modding API documented at hutlihutgames.github.io/void_crew_modding_documentation — StatModifiers, Tags, and DynamicModifiers — replace what would otherwise be hand-rolled Harmony work. Stat application uses the native StatMod / IModifierSource pipeline (no per-getter patches); targeting uses native ModTagConfiguration; a small number of signature perks pick up live-state scaling via ModDynamicScalingValue.

1. Tag-narrowed StatMod bundles

ForgeModuleState.BuildMods() emits one StatMod group per module category instead of broadcasting all 20 stat-type IDs to every module. Each group carries a ModTagConfiguration with RequiredTags = [ Module_Category_X ] so the native filter applies it only where it makes sense.

Tag bucketStatTypes covered
Module_Category_WeaponDamage, FireRate, Range, ProjectileSpeed, Accuracy
Module_Category_ShieldShieldMaxHitPoints, ShieldRechargeSpeed, ShieldAbsorption
Module_Category_EngineForwardPower, EnginePower, YawTorque, ElevationPower, StrafePower, JumpChargeSpeed
Module_Category_ReactorPowerProvided, BatteryRechargeAmount
Module_Category_UtilityProcessingSpeed, HealingSpeed, AttractorMaxRange, AttractorPullVelocity

Built-in tag names are resolved at first access via Resources.FindObjectsOfTypeAll<CsTag>() and cached in CsTagRegistry. If a built-in tag fails to resolve (renamed by a game update), the registry logs a warning and falls back to no tag filtering for that bundle — preserves the broadcast behavior rather than silently dropping all upgrades.

2. Forge_Upgraded tag injection (TagsToAdd)

Every Forge-applied StatMod sets TagsToAdd = [ Forge_Upgraded ]. The native StatTagCollection.ApplyModifier path attaches the tag to the target's runtime tag list. Later phases (Phase 4 perks, Phase 7 cursed/signatures) declare modifiers with RequiredLocalTags = [ Forge_Upgraded ] to target only modules that have been pushed past L3 — without re-querying ForgeOverlayTable.

Forge_Upgraded is the mod's own CsTag ScriptableObject, shipped inside Assets/upgrade_forge.metem alongside the Forge prefab.

3. ModDynamicScalingValue for ~2 signature perks (Phase 7)

ModDynamicValue is abstract — the concrete subclass ModDynamicScalingValue : ModDynamicValue<FloatModifier, float> scales a perk's magnitude against live game state (built-in sources: ShipSpeed, ResourceCount, PowerUsage, DefectCount, BreakersFlipped; curves: Linear, Exponential, Diminishing). Since the type is abstract with a LoadDynamicValue(JObject) static factory, PerkPool.cs builds the configuration as a programmatic JObject — no asset-bundle JSON required. 

var dvJson = new JObject {
    ["type"]      = "ModDynamicScalingValue",
    ["scaling"]   = "Linear",       // ValueScalingType enum
    ["source"]    = "ShipSpeed",    // closed set
    ["min_value"] = 0f,
    ["max_value"] = 0.30f,
};
var dynamicValue = ModDynamicValue.LoadDynamicValue(dvJson);
var statMod = new StatMod(new FloatModifier(0f, ModifierType.AdditiveMultiplier, this),
                          StatType.Damage.Id, weaponTagCfg);
statMod.DynamicValue = dynamicValue;
statMod.InitDynamicElements();

v1 dynamic signatures (the rest of the perk roster stays flat):

PerkStatSourceCurveEffect
StalkerDamageShipSpeedLinearUp to +30% damage at top speed
HoarderFireRateResourceCountDiminishing+X% fire rate; diminishing returns past N resources
Out of scope ModDynamicCondition is not used. Maintenance Burden (cursed-perk operational quirks) stays MonoBehaviour-driven — random shutoffs, manual reset interactions, heat ticks — per the design's original intent. Conditions can be revisited in a later content pass if any cursed perk genuinely benefits from native state gating.

Multiplayer Sync — Authoritative Flow

Photon PUN (confirmed). Host arbitrates; clients sync via VoidManager ModMessage (automatic discovery).

  1. Player initiates upgrade at the Forge in-world.
  2. If non-host: client sends PerkRollRequestMessage to host. If host: skip ahead.
  3. Host validates capacity, confirms relics, computes stat mod, rolls perk if slot match. Commits to ForgeMeterController.
  4. Host broadcasts PerkRollMessage to ReceiverGroup.All with target ref, new level, stat-mod summary, perk (or null), slot index, animation seed.
  5. All clients apply the authoritative result to their local overlay; UI plays the animation using the seed.
  6. Late joiner? Host listens on Events.PlayerEnteredRoom and sends ForgeStateSyncMessage with full overlay state.
  7. Host migration: on Events.MasterClientSwitched, the new host becomes authoritative immediately; outstanding requests time out.
