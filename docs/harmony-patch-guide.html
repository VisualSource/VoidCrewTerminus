HarmonyPatch — Modder's Guide

Harmony 2.x  |  Target: net472  |  Applies to VoidCrewTerminus via Harmony.CreateAndPatchAll in BepinPlugin.Awake.

Overview

Harmony is a runtime-patching library for .NET that lets you change the behaviour of methods you don't own — including game code that ships as compiled assemblies. Instead of editing the game's DLLs (which would break with every patch the developer ships), you write small patch methods that Harmony injects around the original at runtime.

In this project, Harmony is brought in via VoidManager / BepInEx and activated in one line:

VoidCrewTerminus/Plugins.cs — entry point
Harmony.CreateAndPatchAll(Assembly.GetExecutingAssembly());

That single call scans the assembly for every class annotated with [HarmonyPatch] and wires up the patch methods inside them. After that, anytime the game calls the targeted method, your patch runs.

Mental model: Harmony does not edit DLLs on disk. It rewrites the JIT-compiled method body in memory using a stub that calls your Prefix, the original, and your Postfix in sequence. From the game's perspective, the method just behaves differently.

How it works

You write a patch class annotated with [HarmonyPatch]. Inside it, you add static methods named Prefix, Postfix, Transpiler, or Finalizer — Harmony recognizes them by name and slots them into the right place.

Minimal patch example
using CG.Client.UI;
using HarmonyLib;

namespace VoidCrewTerminus
{
    [HarmonyPatch(typeof(FadeController), "Start")]
    internal class FadePatch
    {
        static void Postfix()
        {
            BepinPlugin.Log.LogInfo("FadeController.Start finished");
        }
    }
}

That's the entire pattern. The attribute tells Harmony what to patch, the static Postfix method is your code, and Harmony does the wiring at startup.

Patch lifecycle

For a single call to the patched method, here is the order in which patches run:

Prefix runs first — can skip the original by returning false.
↓
Original runs unless any Prefix returned false.
↓
Postfix runs after the original — sees the result, can modify it.
↓
Finalizer runs last — like a finally, observes / suppresses exceptions.

Transpilers sit outside this sequence — they don't run at call time. They run once when Harmony first compiles the patched method and rewrite its IL instructions.

Multiple patches: Many mods can patch the same method. Prefixes run in priority order; Postfixes also stack. Use [HarmonyPriority] and [HarmonyBefore] / [HarmonyAfter] if you need to control ordering relative to other mods.

Patch types

Prefix

A Prefix runs before the original. Use it to inspect or modify the arguments, to short-circuit the original entirely, or to record state for the matching Postfix.

Prefix static void / static bool Prefix(...)
Return typeMeaning
void Inspect / modify args. Original always runs.
bool Return true to continue, false to skip the original and all later prefixes. Postfix still runs.
Prefix: modify an argument before the original sees it
[HarmonyPatch(typeof(OrbitObject), nameof(OrbitObject.ApplyHitDamage))]
internal class DamageScalerPatch
{
    static void Prefix(ref float damage)
    {
        damage *= 0.5f; // halve incoming damage for everyone
    }
}
Prefix: skip the original entirely
[HarmonyPatch(typeof(OrbitObject), nameof(OrbitObject.Kill))]
internal class GodModePatch
{
    static bool Prefix(OrbitObject __instance)
    {
        if (__instance.Faction == 0) return false; // suppress death for player faction
        return true;
    }
}
Skipping is contagious: Returning false from your prefix prevents the original from running, which can break other mods that expected its side effects (events, state mutations, Photon sync). Use it sparingly and prefer transpilers when you only need to tweak part of the behaviour.

Postfix

A Postfix runs after the original. Use it to react to a completed call, modify the return value, or initialize state on a freshly-constructed object.

Postfix static void Postfix(...)
PatternUse case
Modify __result Take a ref parameter to __result and overwrite it.
Pass-through return A Postfix may declare a return value of the same type — Harmony chains it as the new return.
Side effects Logging, registering events, applying buffs after construction.
Postfix: log every NPC instantiation
[HarmonyPatch(typeof(NpcAI), "OnPhotonInstantiate")]
internal class NpcSpawnLogPatch
{
    static void Postfix(NpcAI __instance)
    {
        BepinPlugin.Log.LogInfo($"NPC spawned: {__instance.gameObject.name}");
    }
}
Postfix: rewrite a return value
[HarmonyPatch(typeof(WeaponModule), "get_CooldownDuration")]
internal class FasterFiringPatch
{
    static void Postfix(ref float __result)
    {
        __result *= 0.75f; // 25% faster fire rate
    }
}

Transpiler

A Transpiler rewrites the original method's IL instructions at patch time. It receives an IEnumerable<CodeInstruction> and must return one. Use it when you need to surgically change a constant, replace a method call, or insert logic mid-method without dropping the rest.

Transpiler skeleton
[HarmonyPatch(typeof(SomeClass), "SomeMethod")]
internal class ConstantPatch
{
    static IEnumerable<CodeInstruction> Transpiler(IEnumerable<CodeInstruction> instructions)
    {
        foreach (var ins in instructions)
        {
            // Replace any "ldc.r4 1.0" with "ldc.r4 2.0".
            if (ins.opcode == OpCodes.Ldc_R4 && (float)ins.operand == 1f)
                yield return new CodeInstruction(OpCodes.Ldc_R4, 2f);
            else
                yield return ins;
        }
    }
}
Transpilers are powerful but fragile: they reference IL opcodes that may shift across game updates. Prefer Prefix/Postfix where possible; reach for a Transpiler only when you need to change behaviour in the middle of a method that you can't otherwise hook.

Finalizer

A Finalizer behaves like a finally block wrapped around the original. It runs whether or not an exception was thrown, can observe the exception via the __exception parameter, and can suppress it by returning null.

Finalizer: swallow an exception so the game doesn't crash
[HarmonyPatch(typeof(SomeFlakyClass), "Update")]
internal class SafeUpdatePatch
{
    static Exception Finalizer(Exception __exception)
    {
        if (__exception != null)
            BepinPlugin.Log.LogWarning($"Suppressed: {__exception}");
        return null; // swallow it
    }
}
Don't silence everything: swallowing exceptions in a Finalizer can mask real bugs. Only use it when you've confirmed the underlying error is benign and you want to prevent a crash.

Targeting: forms of [HarmonyPatch]

The [HarmonyPatch] attribute can be stacked or combined in several ways. The most common forms:

attribute [HarmonyPatch] — common forms
FormUse case
[HarmonyPatch(typeof(T), "MethodName")] Most common. Patches the named method on type T.
[HarmonyPatch(typeof(T), nameof(T.Method))] Refactor-safe: renaming the method breaks the build, not the runtime.
[HarmonyPatch(typeof(T), MethodType.Constructor)] Patches the constructor(s) of T.
[HarmonyPatch(typeof(T), MethodType.Getter, nameof(T.Prop))] Patches the get_Prop accessor. Setter for set_Prop.
[HarmonyPatch(typeof(T), "Method", new[]{ typeof(int), typeof(string) })] Disambiguates overloads by parameter types.

Overloads & properties

When a method has overloads, you must specify the parameter types explicitly — otherwise Harmony throws "ambiguous match":

Disambiguating an overloaded method
[HarmonyPatch(
    typeof(SomeType),
    "DoThing",
    new[] { typeof(int), typeof(OrbitObject) })]
internal class DoThingIntOrbitPatch { ... }

For properties, target the auto-generated accessor by name (get_X / set_X) or use the MethodType.Getter / MethodType.Setter form:

Patching a property setter
[HarmonyPatch(typeof(OrbitObject), MethodType.Setter, nameof(OrbitObject.HitPoints))]
internal class HitPointsSetPatch
{
    static void Postfix(OrbitObject __instance, float value)
    {
        BepinPlugin.Log.LogInfo($"{__instance.name} HP set to {value}");
    }
}

Generics & nested types

Patching generic methods or nested types is possible but verbose — you must reference the closed generic type. For nested types, use the + syntax in Type.GetType or reference them directly:

Targeting a nested type
[HarmonyPatch(typeof(Outer.Inner), "Method")]
internal class InnerPatch { ... }

Injected parameters

Harmony recognizes specially-named parameters in your patch method signatures and fills them in. You only declare the ones you need.

injected Special parameter names
NameTypeMeaning
__instance the target type this of the patched method (instance methods only).
__result method return type The original's return value. Make it ref to overwrite.
__state your choice A value passed from Prefix → Postfix. Set in Prefix as out, read in Postfix.
__originalMethod MethodBase Reflection handle to the original, useful in shared patches.
__exception Exception Finalizer only — the exception thrown by the original (or null).
__args object[] All arguments as a boxed array.
___fieldName field type Access a private field of the target instance. Three underscores + field name.
same-named parameter argument type Declare a parameter with the same name as an original argument to read/modify it.

Passing __state from Prefix to Postfix

A common pattern: snapshot something in the Prefix and react to the change in the Postfix. Harmony auto-threads a __state parameter between them — typed however you like.

__state: detect when HP changes across a method call
[HarmonyPatch(typeof(OrbitObject), nameof(OrbitObject.ApplyHitDamage))]
internal class HpDeltaPatch
{
    static void Prefix(OrbitObject __instance, out float __state)
    {
        __state = __instance.HitPoints; // snapshot BEFORE damage
    }

    static void Postfix(OrbitObject __instance, float __state)
    {
        float delta = __state - __instance.HitPoints;
        if (delta > 0)
            BepinPlugin.Log.LogInfo($"{__instance.name} lost {delta} HP");
    }
}
Accessing a private field with three underscores
[HarmonyPatch(typeof(HullDamageController), "OnDamageReceived")]
internal class HullPrivateFieldPatch
{
    // ___accumulatedDamage maps to the private field "accumulatedDamage"
    static void Postfix(ref float ___accumulatedDamage)
    {
        ___accumulatedDamage *= 0.9f; // scale down accumulation directly
    }
}

Registering patches

In this project, all patches are auto-discovered by a single CreateAndPatchAll call in the plugin entry point. You don't have to manually register each patch class — just decorate it with [HarmonyPatch] and Harmony will find it.

VoidCrewTerminus/Plugins.cs — what already runs at startup
public class BepinPlugin : BaseUnityPlugin
{
    public static ManualLogSource Log;

    private void Awake()
    {
        Log = Logger;
        Harmony.CreateAndPatchAll(Assembly.GetExecutingAssembly());
        Log.LogInfo("VoidCrewTerminus loaded");
    }
}

That means adding a new patch is one step: drop a new class file in the project with [HarmonyPatch(...)] on it. Rebuild, drop the DLL in BepInEx, done.

Convention: Keep patch classes internal and group related patches in one file. Match the existing VoidCrewTerminus/Patch.cs pattern as a starting point.

Common pitfalls

  • Forgetting static: Patch methods must be static. Harmony will silently skip non-static patch methods.
  • Wrong method name spelling: Strings like "Start" aren't checked at compile time. Prefer nameof(...) where possible.
  • Ambiguous overloads: If the target method is overloaded, pass an explicit parameter-type array in [HarmonyPatch] or you'll get a runtime error.
  • Patching throw null stubs directly: VoidCrew.GameLibs ships stubbed bodies. Patching them is fine; calling them from your own non-patch code is not — they'll throw NullReferenceException. See gamelibs-guide.html.
  • Skipping Photon side effects: A Prefix that returns false can silently desync clients if the original sent an RPC. Verify what state the original mutates before suppressing it.
  • Holding instance references: Don't cache __instance in a static dictionary without a cleanup hook — destroyed Unity objects will still satisfy != null in some contexts (the "fake null") and leak memory.

Debugging

When a patch doesn't fire or behaves wrong, the first thing to check is whether Harmony actually attached it. A few techniques:

Log on first run to confirm the patch is wired up
[HarmonyPatch(typeof(FadeController), "Start")]
internal class FadePatch
{
    static bool _logged;

    static void Postfix()
    {
        if (!_logged)
        {
            BepinPlugin.Log.LogInfo("FadePatch attached and firing");
            _logged = true;
        }
    }
}

Other useful techniques:

  • Inspect everything Harmony has patched at startup with Harmony.GetAllPatchedMethods() and log the result — confirms your patch class was discovered.
  • Set HarmonyFileLog.Enabled = true before patching to dump generated IL to disk — invaluable when a Transpiler isn't producing the right code.
  • Watch BepInEx's console (BepInEx/LogOutput.log) — exceptions inside patch methods are logged there, not surfaced to the user.
  • If Harmony.CreateAndPatchAll throws, the message names the offending patch class and usually the reason (ambiguous match, missing method, signature mismatch).
Hot reload: Harmony does not unpatch when you reload the mod. Use harmony.UnpatchSelf() in a custom unload hook if you need clean reload behaviour during iteration.
