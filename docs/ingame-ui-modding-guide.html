In-Game UI Modding — Modder's Guide

BepInEx 5  |  VoidManager 1.2.10  |  Harmony 2.x  |  Target: net472  |  Worked example: hiding the helm TacticalMap.

Overview

Void Crew's UI is Unity-driven and ships in compiled assemblies. To modify it, you don't edit the game's DLLs — you write small patch methods that Harmony injects around the originals at runtime, optionally driven by a runtime toggle exposed in VoidManager's settings panel.

This guide walks the full toolkit available in VoidCrewTerminus, then uses one concrete goal as the worked example: hiding the helm tactical map. The same pattern scales to any other UI element you want to remove, restyle, or extend.

Sibling doc: If you want a deeper dive on Harmony itself — patch types, injected parameters, transpilers, common pitfalls — read harmony-patch-guide.html alongside this one.

The mod toolkit

These are the building blocks already wired into the project. Reach for whichever fits the job.

Mechanisms available
Mechanism Where it lives Use it for
Harmony patches HarmonyLib, applied by Plugins.cs in BepinPlugin.Awake() Modifying existing game UI — Pre/Postfix on Unity lifecycle methods (Awake, Start, OnEnable).
ModSettingsMenu VoidManager.CustomGUI — stubbed at UI/ModSettingsMenu.cs F5 in-game settings panel for your mod. IMGUI-based; subclass and override Draw().
GUITools VoidManager.Utilities DrawCheckbox, DrawSlider, DrawTextField, DrawChangeKeybindButton — pre-built IMGUI helpers.
Messaging VoidManager.Utilities.Messaging Notification(...) / Echo(...) — chat-style on-screen messages.
RuntimeAssetsAPI RuntimeAssets — wired via AssetLoader.cs Loading custom Unity prefabs from .metem asset bundles. For adding UI, not hiding.
ConfigFile BepInEx.Configuration — exposed as Config on BaseUnityPlugin Persistent settings written to BepInEx/config/<guid>.cfg. The state your patch reads at runtime.

Choosing an approach

Match the mechanism to the goal. The right starting point depends on whether the UI already exists in the game and what you want to do with it.

Hide existing Harmony Postfix on the element's OnEnable → gameObject.SetActive(false).
↓
Restyle existing Harmony Postfix mutating RectTransform / Image / Text fields after the game sets them up.
↓
Inject into existing Harmony Postfix on the panel's Start/Awake adding children under __instance.transform.
↓
Add brand-new HUD Load a prefab via RuntimeAssetsAPI, instantiate it under a Canvas, drive it from a MonoBehaviour.
↓
Settings panel Subclass VoidManager.CustomGUI.ModSettingsMenu and draw IMGUI controls.

For our worked example — hiding the helm tactical map — the answer is the first row: a Postfix that deactivates the GameObject, gated by a config toggle.

Station UI: target classes

The game ships station UI under CG.Client.UI.Terminals. These are the candidate types — picked from VoidCrew.GameLibs 1.3.0 / Assembly-CSharp.dll — that a Harmony patch can target.

PRIMARY Top-level controllers
CG.Client.UI.Terminals Root terminal UI controller. Broadest target — patch here to affect every terminal.
CG.Client.UI.Terminals.TacticalMap Tactical map display at the helm station. The worked example below targets this one.
PER-PANEL Visible UI elements at a station
CG.Client.UI.Terminals.HealthBarElement Ship health / status readout widget.
CG.Client.UI.Terminals.RepeatTexturedElement Repeating-texture decorative panel.
CG.Client.UI.Terminals.TacticalMap.RadialFillVE Radial progress / gauge inside the tactical map.
ADJACENT Player HUD (not station UI, but often modified together)
CG.Client.Player.HUD.WeaponControlFrame Weapon control display panel (turret operator HUD).
CG.Client.Player.HUD.AmmoReloadGizmo Ammo / reload indicator.
CG.Client.Player.HUD.ReticleChargeUp Charge-up progress indicator for chargeable weapons.
Don't patch this to hide UI: CG.Client.Ship.Modules.Mediators.TerminalModuleMediator wires terminal UI to gameplay. Disabling it will break interaction (you'll see the panel but can't use the station, or vice versa). Patch the visual element directly instead.

Stubs vs. real DLL

The VoidCrew.GameLibs NuGet package ships stub assemblies — they expose type and method signatures so your code compiles, but the method bodies are empty. That's enough to write a Harmony patch, but not enough to know which lifecycle method actually does the work you want to hook.

Before writing a patch: open the real Assembly-CSharp.dll from <Void Crew install>/Void Crew_Data/Managed/ in dnSpy or ILSpy, navigate to your target class, and confirm which method to hook. Stubs identify what exists; the real DLL tells you how it behaves.

What you're looking for in the decompiled class:

  • What does it inherit from? MonoBehaviour means OnEnable/SetActive apply. VisualElement (UI Toolkit) means you'll be setting style.display instead.
  • Which method first makes the element visible? OnEnable usually. Sometimes Refresh, Show, or a custom name.
  • Exact method signature — parameter types and return type. Harmony needs to match it.
  • Visibility: if the method is private / internal, use AccessTools.Method(typeof(...), "Name") instead of nameof.

Worked example — hide the tactical map

The full pattern is four small pieces: a config holder, a Harmony patch that reads it, a one-line wire-up in Plugins.cs, and a toggle in the existing settings menu. Once this is in place, every additional UI element you want to hide is just one more ConfigEntry<bool> and one more patch class.

1. Config holder

Centralize all your mod's runtime toggles in one place. Using BepInEx.Configuration.ConfigEntry gives you free persistence to BepInEx/config/<guid>.cfg and a value the patch can read on every hook.

VoidCrewTerminus/TerminusConfig.cs
using BepInEx.Configuration;

namespace VoidCrewTerminus;

internal static class TerminusConfig
{
    internal static ConfigEntry<bool> HideTacticalMap;

    internal static void Init(ConfigFile cfg)
    {
        HideTacticalMap = cfg.Bind("StationUI", "HideTacticalMap", false,
            "Hide the tactical map display at the helm station.");
    }
}

2. Harmony patch

A Postfix on OnEnable runs every time the element is shown. We check the config and deactivate the GameObject if the user has opted in. Because the Postfix runs after the original, the game's own setup and event wiring complete first — we only hide the final result.

VoidCrewTerminus/Patches/HideTacticalMapPatch.cs
using CG.Client.UI.Terminals;
using HarmonyLib;

namespace VoidCrewTerminus.Patches;

[HarmonyPatch(typeof(TacticalMap), "OnEnable")]
internal static class HideTacticalMapPatch
{
    static void Postfix(TacticalMap __instance)
    {
        if (TerminusConfig.HideTacticalMap.Value)
            __instance.gameObject.SetActive(false);
    }
}
Why Postfix, not Prefix-returning-false? A Prefix that returns false skips the original entirely — but the original is where the game wires up event subscriptions, caches references, and initializes child elements. Skipping it can break the station's interaction logic even though you only wanted to hide the visual. Postfix lets all that setup complete; you simply deactivate the result.

3. Wire-up in Plugins.cs

Initialize the config before CreateAndPatchAll runs, so the patch has a valid ConfigEntry to read on its first invocation.

VoidCrewTerminus/Plugins.cs — inside BepinPlugin.Awake()
Log = Logger;
TerminusConfig.Init(Config);  // ← add this
Harmony.CreateAndPatchAll(Assembly.GetExecutingAssembly());
AssetLoader.TryLoadAssetBundlesNextToDLL();
Logger.LogInfo($"Plugin {MyPluginInfo.PLUGIN_GUID} is loaded!");

4. Expose the toggle in the settings menu

Replace the placeholder ToggleBool in TerminusModMenu with a binding to the config entry. The user can flip it at runtime; the next time they enter the helm station, the patch sees the new value.

VoidCrewTerminus/UI/ModSettingsMenu.cs — Draw()
public override void Draw()
{
    Label("Terminus");

    TerminusConfig.HideTacticalMap.Value =
        Toggle(TerminusConfig.HideTacticalMap.Value, "Hide tactical map");
}
Scaling out: For each additional UI element you want to hide, add one more ConfigEntry<bool> in TerminusConfig, one more [HarmonyPatch] class in Patches/, and one more Toggle line in Draw(). No central registry needed — Harmony auto-discovers every patch class on assembly load.

Verification

End-to-end check that the toggle actually does what it claims.

  • Build: dotnet build from the repo root. Confirm clean build, no warnings on the new files.
  • Deploy: copy VoidCrewTerminus/bin/Debug/net472/VoidCrewTerminus.dll to <Void Crew install>/BepInEx/plugins/VoidCrewTerminus/.
  • Launch: start Void Crew. Tail BepInEx/LogOutput.log — you should see the Plugin ... is loaded! line and no Harmony exceptions.
  • Toggle: press F5 → open the Terminus mod panel → flip Hide tactical map on.
  • Sit at helm: the tactical map should be hidden. Stand up, re-enter the station — still hidden (patch runs on each OnEnable).
  • Toggle off: flip the setting off, re-enter — map returns.
  • Persistence: restart the game; the setting should persist (it's written to BepInEx/config/<guid>.cfg).

Pitfalls & next steps

  • Don't patch TerminalModuleMediator to hide things. It owns the interaction wiring that connects the player's terminal input to gameplay modules. Disabling it breaks usability even when the visual disappears as expected. Always patch the visual element directly.
  • UI Toolkit elements aren't GameObjects. If the target inherits from VisualElement rather than MonoBehaviour, gameObject.SetActive(false) won't apply (there's no GameObject). Set __instance.style.display = DisplayStyle.None in the Postfix instead.
  • Internal / private targets need AccessTools. [HarmonyPatch(typeof(T), nameof(T.Method))] only works for public members. For private methods use the string-literal form or AccessTools.Method(typeof(T), "Method").
  • Multiplayer: VoidManagerPlugin.MPType = MultiplayerType.All means every player has the mod, but each runs their own patch with their own config. Hiding UI is a local-view change — no networking required.
  • Logging: use BepinPlugin.Log.LogInfo(...) liberally inside the Postfix while you're iterating. Once it works, drop the logs — they spam every OnEnable.
