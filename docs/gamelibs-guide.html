VoidCrew.GameLibs — Modder's Guide

Assembly-CSharp.dll · v1.3.0  |  Target: net472  |  Decompiled stubs — bodies are throw null; patch, don't call directly.

Overview

VoidCrew.GameLibs is the NuGet package that bundles all of Void Crew's game assemblies as reference stubs. The key game-logic DLLs are:

  • Assembly-CSharp.dll — all game gameplay code
  • VoidCrewCommon.dll — shared utilities, common types
  • ResourceAssets.dll — asset reference types (CloneStarObjectRef, GUIDUnion)

Because these are decompiled stubs, every method body is throw null. You interact with the game via Harmony patches — you never instantiate or call these types directly; you hook into the game's own call stack.

Multiplayer authority: Void Crew uses Photon PUN2. Most mutating operations only run on the master client (room host). Before writing state in a patch, check PhotonNetwork.IsMasterClient or the object's .AmOwner property.
Stubs throw null: Method bodies are stubs. Calling base.SomeMethod() from a Transpiler or calling game methods from your own non-patch code will throw at runtime. Patches work because Harmony replaces the body before it executes.

Enemy System

Enemy ships in Void Crew are OrbitObjects — the base for every space entity. On top of OrbitObject, NPC ships carry an NpcAI component that drives behaviour. The AIDirector singleton owns a list of Spawner instances; each Spawner manages a pool of SpawnerUnitEntry groups and decides when to instantiate new enemies via ObjectFactory.

The lifecycle looks like this: 

// Simplified flow (not actual source)
// 1. AIDirector ticks each Spawner on an interval
// 2. Spawner picks a SpawnerUnitEntry that fits current intensity budget
// 3. SpawnerUnitEntry contains SpawnerUnitDef objects (each = an enemy prefab + intensity cost)
// 4. ObjectFactory.InstantiateSpaceObjectByGUID spawns the prefab over Photon
// 5. IPunInstantiateMagicCallback.OnPhotonInstantiate fires on NpcAI → wires up behaviours

Once alive, the NpcAI runs a simple state machine driven by Behaviours (e.g. BehAttack, BehOrbitAroundTarget), Commands (external instructions queued on the AI), and Signals (event messages like SignalHitFromSource). The active behaviour is selected by the SignalResolver field.

Patch stubs

Hook new enemy on spawn to modify its stats or aggro:

EnemySpawnPatch.cs — postfix NpcAI.OnPhotonInstantiate
using Gameplay.NPC.AI;
using CG.Space;
using HarmonyLib;
using Photon.Pun;

[HarmonyPatch(typeof(NpcAI), "OnPhotonInstantiate")]
internal class NpcAISpawnPatch
{
    static void Postfix(NpcAI __instance, PhotonMessageInfo info)
    {
        // Runs on ALL clients after the NPC prefab is instantiated over Photon.
        var orbitObj = __instance.GetComponent<OrbitObject>();
        if (orbitObj == null) return;

        // Example: scale up max HP for all NPCs
        orbitObj.MaxHitPoints.BaseValue *= 1.5f;
        orbitObj.hitPoints = orbitObj.MaxHitPointsValue;
    }
}

Inject a new spawn group into the default spawner at session start:

CustomEnemyInjector.cs — postfix AIDirector.OnEnable
[HarmonyPatch(typeof(AIDirector), "OnEnable")]
internal class AIDirectorPatch
{
    static void Postfix(AIDirector __instance)
    {
        // Only the master client controls spawning.
        if (!PhotonNetwork.IsMasterClient) return;

        // Build a unit def pointing at your custom enemy prefab.
        // Replace the GUIDUnion with your actual asset GUID.
        var customDef = new SpawnerUnitDef(
            intensity: 2,
            guid: new GUIDUnion("your-asset-guid-here"),
            path: "Enemies/Lech"
        );

        // Wrap in an entry and add to the default spawner.
        var entry = new SpawnerUnitEntry();
        entry.UnitDefs.Add(customDef);
        __instance.defaultSpawner?.AddSpawnerUnitEntry(entry);
    }
}

Ship Modules & Hull Damage

The player ship is a PlayerControlledShip (extends OrbitObject) built from a ShipHull containing rooms and ModSocket sockets. Each socket can hold a CellModule subclass — weapons, shields, engines, relics, etc.

Hull damage is tracked separately from hit points. HullDamageController listens to the ship's DamageReceived event, accumulates damage, and creates HullBreach objects (minor or major) when thresholds are crossed. Breaches must be manually repaired by players.

Relics

Relics are CarryableObjects that slot into ModSocket sockets. The relic shrine module (RelicShrineModule) configures the sockets on the ship and resolves which OrbitObject stat effects to apply when a relic is inserted.

The RelicSocketController is the networked glue: it listens to ICarryablesSocket events (OnRelicAdded / OnRelicRemoved) and serializes socket state to late-joining clients via OnPhotonSerializeView. The DestroyRelicTrigger field is a ChangeResponsiveBool that, when flipped, starts a coroutine to remove and destroy the carried relic.

Replication note: Relic state is serialized in RelicSocketController.OnPhotonSerializeView. To disable replication of relic state to other clients, patch that method. Be aware this means late-joiners won't see which relics are socketed.

Photon Networking & Replication

Void Crew uses Photon PUN2 with a master-client authority model. The master client is the room host and owns the simulation: spawning enemies, driving the AIDirector, managing sector transitions.

Game objects that need to be synchronized extend MonoBehaviourPun and implement IPunObservable (OnPhotonSerializeView). New network objects are created through ObjectFactory — not directly via PhotonNetwork.Instantiate — so that the game's GUID-based asset registry is used instead of resource paths.

Late-joiners receive state via IPunInstantiateMagicCallback.OnPhotonInstantiate plus the serialized stream from OnPhotonSerializeView. Some types (like NpcAI) split their serialization across IPunSplitSerialize to avoid single-stream size limits.

Master-client guard: Any mutation of shared state must be gated on PhotonNetwork.IsMasterClient. Non-master clients should read state, not write it.

Patch stubs

Intercept all networked object instantiation to inject custom data (e.g., level):

SpawnDataPatch.cs — prefix ObjectFactory.CheckForLevelUpdate
[HarmonyPatch(typeof(ObjectFactory), "CheckForLevelUpdate")]
internal class SpawnLevelPatch
{
    static void Prefix(GUIDUnion guid, ref Dictionary<byte, object> instantiationData)
    {
        // Runs before the game injects its level data.
        // Add your mod's custom byte-keyed data into instantiationData here.
        // Keys 0-127 are used by the game; use 200+ for mod data to avoid collisions.
        instantiationData ??= new Dictionary<byte, object>();
        instantiationData[200] = MyModState.GetLevelForGuid(guid);
    }
}
