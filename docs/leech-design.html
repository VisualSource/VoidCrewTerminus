Leech Carrier Encounter — System Design

v1 design VoidCrewTerminus BepInEx + Harmony + VoidManager

Technical design for the Leech Carrier encounter — a new deep-sector combat archetype that forces EVA mid-combat. Extends the Upgrade Forge's Sector Escalation system. For the higher-level proposal, see leech-rfc.html.

Context

The Upgrade Forge introduces DifficultyScalar on ForgeMeterController — a per-sector counter that scales enemy stats and loot tier. The scalar is a clean numeric handle, but the Forge's Phase 6 only uses it to dial existing parameters: enemy HP, enemy damage, drop-tier biasing. It doesn't introduce any new encounter archetypes.

The TODO file at the project root names a wish: "lech — launched via missiles, attaches to ship and causes damage to ship until removed by a player like Star Wars buzz droids — small robotic spider." This design realizes that wish as an encounter gated and scaled by DifficultyScalar: a vanilla enemy class becomes a Leech Carrier at scalar ≥ 2, firing Leech Missiles that deploy Leech parasites on the player's hull on impact. Players must EVA to remove them. The Forge already defined the scalar; we plug into it.

Why a new archetype rather than tuning existing enemies Late-sector Void Crew runs feel numerically harder but mechanically identical to early runs — the player still plays the same cockpit/gunnery/reactor loop, just with bigger numbers on both sides. This encounter changes what crew is doing in deep sectors: someone must leave their station and EVA. It's a deliberate gameplay shape, not a stat bump.

System Summary (one paragraph)

At DifficultyScalar ≥ 2, a vanilla enemy class (identified pre-flight) is patched to fire a homing-with-finite-turn-rate Leech Missile. The missile is interceptable by any weapon fire and engaged by vanilla point-defense (with resistance scaling per scalar). On hull impact, no hull damage — instead, 1–3 Leech instances spawn at the impact point. Each anchors at the nearest hull patch and is assigned a variant deterministically based on proximity to a module: near a module → Module-Biter (B-type), far → Hull-Biter (A-type). B-type applies a non-stacking effectiveness debuff plus stacking module HP damage; A-type damages hull section HP on a bite cadence. Removal is EVA-only: aim the existing multi-tool at the Leech, hold to remove for 3–5s, complete 1–3 periodic input prompts. Miss a prompt → the Leech escapes, crawls to a new random hull location, re-anchors. Capped at one escape per Leech. Total concurrent Leeches on the ship capped at 8. If a B-type's targeted module is destroyed (by it or by other damage), the Leech self-destructs ("mission complete"). All state is host-arbitrated and synced via VoidManager ModMessage, mirroring the Upgrade Forge's pattern. Player awareness reuses vanilla observation tools (telescope module, cockpit 3rd-person, gunner external view) — no bespoke hull-status UI. Audio cues are diegetic through the hull, plus one ship-wide resonance sting on first impact per encounter.

Glossary

Leech Carrier (Host)
A vanilla enemy ship class patched to carry the Leech Missile weapon at DifficultyScalar ≥ 2. The specific class is identified pre-flight.
Leech Missile
The deploying projectile. Homing with finite turn rate, killable by any weapon fire, engaged by vanilla point-defense with per-scalar HP resistance. Does no hull damage on impact — payload is the Leeches.
Leech
A small robotic parasite that attaches to the player ship's hull. Two variants, deterministically assigned by spawn-location proximity to a module.
Module-Biter (B-type)
Default Leech variant. Anchors near a module, applies a non-stacking effectiveness debuff to that module while attached, and deals stacking HP damage on a tick. Self-destructs if its targeted module is destroyed.
Hull-Biter (A-type)
Rarer Leech variant. Anchors on bare hull, deals damage to the hull section it's on via a visible "chomp" bite cadence. Vanilla breach/depressurization mechanics handle consequences.
DifficultyScalar
Defined in the Upgrade Forge design. Integer counter on ForgeMeterController, increments by +1 per successful sector jump. This encounter gates at scalar ≥ 2 and uses scalar tiers (2–3, 4–5, 6+) for parameter tuning.
Containment Failure
The event where an EVA-removing player misses a containment input prompt and the Leech escapes to a new hull location. One per Leech max.
Hold-to-Remove
The 3–5 second EVA interaction in which a crewmember aims the multi-tool at a Leech and holds activation. Periodic prompts test containment.
Containment Prompt
A brief input cue (Void Crew's existing boot-panel-style mini-prompt) shown during hold-to-remove. Player must respond within a window (1.0–1.5s) or trigger Containment Failure.
Concurrency Safety Rail
Hard cap of 8 concurrent Leeches on the player ship. When hit, subsequent missile impacts trigger VFX but spawn no Leeches.
Diegetic Awareness
The design principle that crew learn of Leeches through the game world (visual observation via telescope/cockpit/turret views; diegetic audio through the hull) rather than through mod-side alert popups or UI overlays.

Host Ship

A vanilla enemy class is patched at DifficultyScalar ≥ 2 to fire the Leech Missile in addition to (or in place of) its standard weapon, on a per-scalar fire interval. The specific class is a pre-flight identification item — chosen so it already appears in deep sectors, has a recognizable silhouette, and has a weapon firing pattern we can hook into.

The patch surface (target class TBD via pre-flight):

  • Hook the host enemy's per-tick AI / weapon-firing method (Harmony postfix).
  • If DifficultyScalar < 2: no behavior change.
  • If DifficultyScalar ≥ 2 and fire-interval cooldown elapsed: spawn a Leech Missile at the host's weapon mount, aimed at the player ship.
  • Fire-interval cooldown is per-host-instance, scaled by DifficultyScalar (30s → 25s → 20s across scalar tiers).

Leech Missile

The projectile is a new MonoBehaviour spawned at host weapon mount. It is mod-side; vanilla netcode does not replicate it. State sync is via LeechMissileFiredMessage (see Networking).

Guidance

  • Homing with finite turn rate. Each fixed-update, compute the desired-heading vector toward the player ship; rotate the missile's velocity toward it bounded by turnRateRadiansPerSecond × deltaTime.
  • turnRateRadiansPerSecond scales 1.0× → 1.3× → 1.6× across scalar tiers.
  • Constant forward speed in body frame; speed scales 1.0× → 1.15× → 1.3×.

Lifecycle

  1. Host fires the missile via LeechMissileFiredMessage. All clients spawn the local-state mirror.
  2. Each tick: host updates trajectory canonically, broadcasts position deltas as needed (rate-limited; clients dead-reckon between updates).
  3. Collision detection runs host-side. Two outcomes:
    • Intercepted by point-defense or weapon fire — missile takes HP damage; if HP ≤ 0 it explodes, host broadcasts LeechDespawnMessage.
    • Impacts player ship — host triggers spawn logic (see Spawn on Impact) and broadcasts LeechSpawnMessage.

Point-defense integration

The Leech Missile registers itself as a valid point-defense target by exposing whatever interface vanilla projectiles expose (pre-flight item). Resistance is implemented as a high missileHp field that scales per scalar — PD shots tick down HP, and only when HP ≤ 0 is the missile destroyed.

Audio & visual tell

  • Distinct silhouette: elongated capsule (~2× length of vanilla projectiles). v1 placeholder: red-tinted primitive.
  • Audio warble: constant-pitch looping clip on the missile prefab.
  • Vanilla projectile-warning system picks it up automatically (pending pre-flight confirmation); if not, a patch is added to register it.

Spawn on Impact

On player ship collision, the missile triggers the spawn sequence host-side:

  1. Determine the leech count for this missile by rolling on the per-scalar range:
    • Scalar 2–3 → roll 1–2
    • Scalar 4–5 → roll 1–3
    • Scalar 6+ → roll 2–3 (cap 3)
  2. Query the Concurrency Safety Rail: if current active leech count on the player ship is ≥ 8, abort spawn (still play impact VFX). Otherwise clamp roll so total active never exceeds 8.
  3. For each leech to spawn:
    • Pick an anchor point near the impact (jittered within a radius).
    • Project onto the nearest hull surface point (raycast against the player ship's hull mesh).
    • Run the variant check: FindNearestModule(anchorPoint) — if distance ≤ variantProximityThreshold, classify B-type targeting that module. Else classify A-type targeting the hull section.
    • Create a LeechController instance, attach it as a child of the player ship transform at the anchor point.
  4. Broadcast LeechSpawnMessage with the batch (positions, variants, target module IDs or hull-section IDs).

Module-Biter (B-type)

State, per attached B-type:

class LeechControllerBState {
    Guid     LeechId;
    int      TargetModuleInstanceId;
    Vector3  AnchorWorldOffset;     // relative to ship transform
    float    AttachedDurationSec;
    bool     HasEscapedOnce;
    bool     IsBeingRemoved;        // true while a player is holding the multi-tool on it
    float    RemovalProgressSec;
}

Behavior:

  • Debuff application. While any B-type is attached to a module, the module's effectiveness multiplier drops by the per-scalar magnitude (−25% / −40% / −50%). Implemented via the native StatTagCollection pipeline — a singleton LeechModuleDebuffApplicator implements IModifierSource, owns a per-module reference count, and on first attach pushes a category-tagged StatMod bundle into module.Stats via ApplyModifiers. On last detach it calls RemoveModifier(this). No Harmony postfix on effectiveness getters. Non-stacking falls out naturally from the ref count. See Ship-level Leech Exposure for the shared CsTagRegistry / ModuleStatBundles utilities.
  • Module HP damage. Every moduleDamageTickSec (~1.0s), deal moduleDamagePerTick × scalarMultiplier to the target module's vanilla HP. Stacking: each B-type ticks independently. Direct value change — not a stat modifier.
  • Self-destruct trigger. Subscribe to the target module's destruction event (or poll its HP). When the module is destroyed, the leech plays self-destruct VFX/SFX and despawns. Routes through LeechModuleDebuffApplicator.Detach so the ref count decrements correctly. No AOE damage in v1.
  • Tag injection. The applicator's StatMod bundle carries ModTagConfiguration { TagsToAdd = [ Module_Leech_Attached ] }. The native pipeline attaches the tag on first apply and strips it on remove. Other systems (audio, future hooks) filter by RequiredLocalTags = [ Module_Leech_Attached ].

Hull-Biter (A-type)

State, per attached A-type:

class LeechControllerAState {
    Guid     LeechId;
    int      TargetHullSectionId;
    Vector3  AnchorWorldOffset;
    float    AttachedDurationSec;
    float    SecSinceLastChomp;
    bool     HasEscapedOnce;
    bool     IsBeingRemoved;
    float    RemovalProgressSec;
}

Behavior:

  • Bite cadence. Every chompIntervalSec (~2.0s), deal chompDamage × scalarMultiplier to the target hull section's HP. Plays chomp VFX + diegetic audio cue (audible from inside the section).
  • Breach is emergent — vanilla section-HP-zero rules apply. Once the section breaches, the leech keeps chewing on whatever's left. (Vanilla breach repair is unaffected — a leech anchored next to a breach can still be removed by EVA.)

EVA Removal

Removal is the player-side mini-interaction. It runs entirely client-side on the EVA player's machine, with host arbitration of outcomes.

Trigger

  • Player must be in EVA mode (not inside a sealed compartment). Pre-flight: confirm the EVA-detection API.
  • Player aims the existing multi-tool at a Leech (raycast from tool transform).
  • Player holds the multi-tool's activation input.

Hold state machine

  1. On hold start: client sends LeechRemovalTickMessage(START, leechId) to host. Host validates (leech exists, player is in EVA range, leech isn't mid-flee). On success, host marks the leech as IsBeingRemoved = true and broadcasts the update.
  2. Removal progress accumulates on the EVA client. The host receives periodic tick messages and is the canonical authority on remaining duration.
  3. Periodically (host schedules), host broadcasts LeechRemovalPromptMessage — a containment input prompt. The EVA player has the per-scalar response window (1.0–1.5s) to send LeechRemovalTickMessage(PROMPT_RESPONSE, <result>).
  4. If response is in time → continue. If timed out or wrong → trigger Containment Failure (see next section).
  5. When cumulative removal progress reaches the per-scalar duration (3–5s), host broadcasts LeechDespawnMessage and the leech is removed.

Interruption

  • Player releases the multi-tool → host clears IsBeingRemoved. Progress resets (does not persist).
  • Player loses line-of-sight or exits EVA range → host clears IsBeingRemoved. Progress resets.
  • Player suit ruptures / dies → progress resets. Leech remains attached.

Containment Failure

On a missed or incorrect containment prompt response:

  1. Host marks HasEscapedOnce = true on the leech.
  2. Host picks a new random hull anchor point (different from current location, biased away from current EVA player to make the chase meaningful).
  3. Host broadcasts LeechFleeMessage(leechId, oldAnchor, newAnchor, newVariant). The variant is re-rolled at the new anchor by the same proximity check.
  4. Clients animate the crawl from old to new anchor as a simple lerp (v1) over fleeAnimationDurationSec (~2.5s). Damage is paused during this window.
  5. On arrival, the leech re-anchors and resumes damage in its new variant identity.

Subsequent containment attempts on this leech ignore prompt-miss escape — HasEscapedOnce short-circuits the escape rule. The next removal completes normally.

Sector Escalation Scaling

All scaling reads from ForgeMeterController.DifficultyScalar (defined in the Upgrade Forge). A new helper, LeechEscalationScaling, exposes a single API: GetParams(int difficultyScalar) → LeechScalingParams.

The struct:

struct LeechScalingParams {

    int    LeechCountMax;

Lookup is a switch on scalar tier (2–3 / 4–5 / 6+). Values come from TerminusConfig.

Crew Awareness

No bespoke alert UI, no hull-status console. Awareness is composed from vanilla tools + diegetic audio:

  • Vanilla projectile warning picks up the inbound missile (pre-flight to confirm). Plus the missile's distinct silhouette + audio warble.
  • External observation: the ship's telescope module, the pilot's 3rd-person cockpit view, and gunner turret external views all show attached leeches naturally.
  • Diegetic audio:
    • Impact thunk on missile hit (3D-positional from impact location).
    • Chomp clicks for A-type bite cadence (3D-positional, audible inside the affected section).
    • Module stutter audio for B-type debuff (3D-positional from the module).
    • Servo snap on containment failure.
    • Detach pop on successful removal.
    • Thunk-pop on self-destruct.
  • Ship-wide first-impact sting: when the first Leech of an encounter anchors, play a brief subtle hull-resonance hum audible everywhere on the ship. Subsequent leeches in the same encounter do not retrigger the sting.

Safety Rails

  • Concurrent leech cap (8). Hard limit on active leeches on the player ship. Subsequent missile impacts play VFX but spawn no leeches.
  • One escape per leech. A leech with HasEscapedOnce = true cannot escape again. Guarantees finite encounter duration.
  • Removal-progress reset on interrupt. Partial removal does not persist across hold sessions. Forces a full hold-to-remove per attempt.
  • Variant re-roll on flee. A fleeing leech's new variant is determined by its new anchor location (same proximity rule). A B-type can flee and become an A-type and vice versa.
  • Module-destroyed → self-destruct. B-types whose target module is destroyed don't try to retarget; they self-destruct gracefully.
  • Run-end clears all leech state. On ship death, all leech instances and their state records are dropped (matches per-run reset of the Upgrade Forge).

Architecture

The feature lives under VoidCrewTerminus/Leech/ (new directory) and VoidCrewTerminus/Net/ (created by Upgrade Forge Phase 8, reused here).

  • Mod-side overlay, no game-data modification. Same pattern as the Upgrade Forge — no ScriptableObjects modified, no asset-bundle changes to the game install.
  • Host-arbitrated state. All authoritative state lives on the host's LeechEncounterController singleton. Clients hold a mirror that updates on inbound ModMessages.
  • Per-leech MonoBehaviour. Each leech is a LeechController component attached as a child of the player ship transform. Position is in ship-local space so it follows during ship maneuvers.
  • Reuses Upgrade Forge primitives. Reads DifficultyScalar from ForgeMeterController. Hooks into the same per-run reset events. Uses the same ModMessage registration discovery. Reuses the forge's CsTagRegistry and ModuleStatBundles (see Forge Refinements) for tag lookup and per-category stat lists.
  • Native StatModifier pipeline for the B-type debuff. No Harmony postfix on effectiveness getters — the debuff is delivered via module.Stats.ApplyModifiers through a singleton IModifierSource applicator. Reference counting in the applicator delivers the design's non-stacking rule.
  • Ship-level exposure of leech count. The encounter controller emits both tags (qualitative bucketing) and a custom ModDynamicValue / ModDynamicCondition pair (continuous scalar + gating), mirroring how the game exposes BreakersFlipped. See Ship-level Leech Exposure.

Ship-level Leech Exposure

Other mods (and future leech-side effects) should be able to react to how many leeches are currently on the ship through the native modifier pipeline, the same way they can react to BreakersFlipped. The encounter controller exposes the count three ways: qualitative tag bucketing, a continuous scalar dynamic value, and a threshold dynamic condition.

Tag bucketing (qualitative)

LeechEncounterController injects ship-level tags as the active count crosses thresholds. Bucketing transitions remove the old bucket and add the new one in one update.

TagActive when…
Ship_HasLeechescount ≥ 1
Ship_LeechSwarm_Small1 ≤ count ≤ 2
Ship_LeechSwarm_Heavy3 ≤ count ≤ 5
Ship_LeechSwarm_Criticalcount ≥ 6

Other mods declare modifiers with ModTagConfiguration.RequiredTags = [ Ship_LeechSwarm_Heavy ] (or similar) to activate effects only during heavy infestation. Pure native-pipeline filtering, no bespoke hook.

LeechAttachedCountValue (continuous scalar)

A custom ModDynamicValue<FloatModifier, float> subclass that exposes the live attached count as a float source for any modifier's DynamicValue slot. Implements:

  • RecalculateValue() — reads LeechEncounterController.AttachedCount on demand.
  • OnInitialize() / OnDestroy() — registers/unregisters a listener on the controller's AttachedCountChanged event.
  • ValidateType() — declares compatibility with FloatModifier.
  • DynamicValueDescription(...) — tooltip string for inspector / debug output.

Update is push-based: the controller raises AttachedCountChanged on every attach / detach / flee; the value calls inherited InformValueChange(), which fires OnValueChange on each subscribed StatMod. No per-frame polling.

LeechAttachedCountCondition (threshold gating)

A parallel ModDynamicCondition subclass for the condition slot. Constructor takes a threshold and a comparison operator (≥, ≤, ==, etc.). ShouldApply() reads LeechEncounterController.AttachedCount and compares. Honors the inherited InvertApplication flag. Used by modifiers that need binary activation — e.g. "this perk is disabled while any leech is attached".

Construction without JSON

Both subclasses are normal classes; the abstract base has a parameterless constructor. They are instantiated directly in C# — bypassing the static LoadDynamicValue(JObject) / LoadDynamicCondition(JObject) factories. After assignment to statMod.DynamicValue / statMod.DynamicCondition, the helper calls statMod.InitDynamicElements() if pre-flight confirms it's required. A small LeechDynamics static helper centralises the wiring: 

// "shield recharge drops up to 30% as leech count rises"
LeechDynamics.WithLeechCountScaling(statMod, min: 0f, max: 0.30f, curve: Linear);

// "this cockpit perk is disabled while 3+ leeches are attached"
LeechDynamics.GateByLeechCount(statMod, atLeast: 3);
Example consumer Imagine a future "panic mode" cockpit perk that activates when Ship_LeechSwarm_Heavy is set: piloting controls become twitchier (engine power up, accuracy down). Authored as a normal StatMod with RequiredTags = [ Ship_LeechSwarm_Heavy ] — no leech-specific code anywhere. The leech feature exposes the surface; other features consume it.
Out of scope ModDynamicCondition is not consumed by the leech debuff itself (the debuff is unconditional while attached). The leech contributes a condition source for other systems to consume; it doesn't gate its own effects. Tick-based damage paths (B-type module HP damage, A-type hull section chomp) remain direct value changes — they are not stat modifiers.

Multiplayer Sync — Authoritative Flow

Photon PUN via VoidManager ModMessage (automatic discovery). Same pattern as the Upgrade Forge.

Encounter lifecycle

  1. Host's HostShipWeaponPatch postfix fires. Host validates scalar gate and fire-interval cooldown.
  2. Host instantiates a local LeechMissileBehavior and broadcasts LeechMissileFiredMessage with origin, heading, scalar snapshot, missile ID.
  3. Clients receive the message and spawn their mirror missile. From here, host updates missile position periodically (rate-limited; clients dead-reckon between updates).
  4. On host-side collision (interception or player-ship hit), host either despawns the missile (broadcast LeechDespawnMessage for the missile ID) or triggers spawn-on-impact.
  5. Spawn-on-impact: host computes leech batch (count, anchor positions, variants, target IDs). Broadcasts LeechSpawnMessage. Clients instantiate mirror leeches.
  6. Per-tick: host computes leech damage and broadcasts LeechStateUpdateMessage at a sane rate (e.g. once per 0.5s, not every frame).

Removal flow

  1. EVA client detects player aiming multi-tool at a leech; sends LeechRemovalTickMessage(START, leechId) to host.
  2. Host validates: leech exists, isn't mid-flee, EVA player is in range. On success, marks IsBeingRemoved=true and broadcasts a state update.
  3. Host schedules containment prompts at intervals: LeechRemovalPromptMessage(leechId, promptIndex, deadlineTimestamp).
  4. EVA client receives prompt, displays UI; on response, sends LeechRemovalTickMessage(PROMPT_RESPONSE, result). Host validates timing against the deadline.
  5. If response in time → host continues. If timed out or wrong → host transitions to flee (broadcasts LeechFleeMessage).
  6. If hold completes successfully → host broadcasts LeechDespawnMessage.

Late joiner

On VoidManager.Events.PlayerEnteredRoom, the host serializes the current LeechEncounterController state (all active missiles + leeches + their fields) and sends LeechStateSyncMessage directly to the new client. The client reconstructs mirror objects from the snapshot.

Host migration

On VoidManager.Events.MasterClientSwitched, the new host promotes its mirror state to authoritative. Outstanding pending removal-prompt deadlines are extended by a grace window (~1s) to account for the transition. Mirror state is sufficiently complete for promotion because every LeechStateUpdateMessage carries the full per-leech state — no incremental deltas without anchor state.

Pre-Flight Verification

Open Questions

  • Host enemy class. Decided pre-flight. Documented choice must be added to this doc once made.
  • Variant proximity threshold value. The radius defining "near a module" for A/B classification. Starts at config default, refined in playtest.
  • Concurrency cap (currently 8). Playtest may push up or down.
  • Reuse boot-panel UI or author bespoke prompts? Decided in Phase 5 pre-flight; depends on UI element's contextual coupling.

  • Containment prompt input type. Single-button tap by default; could become a directional input or short sequence at deep scalar.
  • First-impact sting per encounter, or per missile? Currently per encounter. Per missile could feel repetitive but might be clearer.
  • Host migration grace window (1s). May need tuning if pending prompts feel like they get dropped during migration.
